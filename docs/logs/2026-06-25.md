## Sesión 1 — Refactorización MCP (alineación con Astro best practices)

### Prompt

Revisar el código del portfolio y refactorizarlo para alinearlo con las mejores prácticas de Astro 5: reemplazar el script inline `is:inline set:html` por el atributo `data-data` en el `<body>`, cambiar los `onclick` por `data-lang` + `addEventListener`, eliminar la función global `window.changeLanguage`, añadir `tsconfig.json` y crear el directorio `.agents/` con la skill de frontend-design.

### Plan

Se identificaron tres problemas principales en el código actual:
1. **Script inline con `set:html`** — Astro desaconseja inyectar HTML serializado mediante `is:inline`. La alternativa correcta es pasar los datos como atributo `data-*` en una etiqueta HTML y leerlos desde el DOM en el cliente.
2. **Gestores de eventos inline (`onclick`)** — Mezcla de lógica JS con marcado HTML. Se migran a `data-lang` + `addEventListener` en `client.js`.
3. **Variable global `window.changeLanguage`** — Contamina el scope global. Se encapsula toda la lógica en `client.js`.

Adicionalmente, se añade `tsconfig.json` (extiende `astro/tsconfigs/base`) para habilitar el type-checking del ecosistema Astro, y se crea el directorio `.agents/` con la skill `frontend-design` y el archivo `skills-lock.json`.

Se descartó la opción de mantener `is:inline` porque Astro ya no lo recomienda para datos dinámicos, y usar fetch() para cargar JSONs en cliente añadía complejidad y latencia innecesarias.

### Cambios

#### `src/layouts/BaseLayout.astro`
- **Eliminado** el bloque `<script is:inline set:html>` que inyectaba `window.DATA = {...}` directamente en el HTML.
- **Añadido** el atributo `data-data` al `<body>` con el valor del objeto `data` serializado mediante el helper `Astro.serialize(data)` (que escapa HTML entities automáticamente).
- El template del layout ahora pasa `data` como prop desde `index.astro` (que recopila todos los JSONs: lang, profile, skills, education, projects, experience, certificates).

#### `src/scripts/client.js`
- **Eliminada** la función `decodeHtml()` — ya no es necesaria porque el browser decodifica automáticamente el contenido del atributo `dataset` al acceder a `document.body.dataset.data`.
- **Lectura de datos**: `const data = JSON.parse(document.body.dataset.data)` — reemplaza `window.DATA`.
- **`changeLanguage()` renombrada a `h()`** — función local encapsulada, ya no expuesta globalmente.
- **Event listeners**: se añadió `document.querySelectorAll("[data-lang]").forEach(el => { el.addEventListener("click", () => h(el.dataset.lang)) })` que reemplaza los `onclick="changeLanguage('es')"` del HTML.
- **Inicialización**: `init()` renombrada a `$()` y llamada al final del script. Carga el idioma guardado de `localStorage`, ejecuta `translateUI()`, activa el botón de idioma correspondiente, y dispara `scrollReveal()`.

#### `src/components/LangSwitcher.astro`
- **Reemplazados** los atributos `onclick="changeLanguage('es')"` y `onclick="changeLanguage('en')"` por `data-lang="es"` y `data-lang="en"`.
- El JS de `client.js` ahora selecciona estos elementos por `[data-lang]` y asigna el evento mediante `addEventListener`.

#### `tsconfig.json` (nuevo)
- Extiende `astro/tsconfigs/base`.
- Configura `strict: true`, `jsx: "preserve"`, `moduleResolution: "bundler"`.
- Permite type-checking de archivos `.astro`, `.ts`, `.tsx`.

#### `.agents/skills/frontend-design/SKILL.md` (nuevo)
- Skill de opencode con instrucciones de diseño frontend: paletas CSS first, tipografía, sistema de espaciado, animaciones con propósito, responsive mobile-first.

#### `skills-lock.json` (nuevo)
- Registro de skills instaladas para opencode: `frontend-design` con hash y fecha.

### Build

```
npm run build
```
Resultado: exitoso. 1 página generada en ~500ms. Sin warnings ni errores.

## Sesión 2 — Refactorización de diseño (aplicación de frontend-design skill)

### Prompt

Aplicar la skill `frontend-design` para mejorar la estética del portfolio: elegir tipografías, refinar la paleta de colores, añadir una firma visual (cuadrícula computacional), reformular el hero como una tesis profesional, eliminar emojis de los títulos y mejorar la jerarquía visual general.

### Plan

Se cargó la skill `frontend-design` y se evaluaron sus recomendaciones punto por punto:

**Tipografía:**
- Se evaluaron varias combinaciones. Se eligió **Space Grotesk** para display (títulos, nombre, badges) por su carácter técnico/geométrico que evoca matemáticas, código y física computacional — encaja perfectamente con el perfil del usuario. **Paleta de acento:**
- Se refinó el azul de acento de `#58a6ff` (GitHub default, genérico) a `#7fc1fe` (eléctrico, más distintivo).
