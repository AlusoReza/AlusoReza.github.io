## Session 1: Fix lang-switcher flash during desktop→mobile transition

### Prompt
When resizing the viewport past 1235px (desktop→mobile), the floating lang-switcher appears momentarily in the sidebar area while the sidebar slides off-screen to the left. User asked to ensure the sidebar animation maintains the same format as before the animation.

### Plan
The root cause was a CSS specificity discontinuity: the base `.lang-switcher-floating { opacity: var(--sidebar-fade) }` correctly makes the switcher invisible at ≤1236px, but the `@media (max-width: 1235px)` rule overrides it with `opacity: 1; transition: none` — creating a flash at the exact frame the sidebar begins its `translateX(-100%)` transition. The `sidebar-locked` mechanism (which forces `--sidebar-fade: 0`) was only applied in the mobile→desktop direction, never when shrinking past 1235px.

Fix: Apply `sidebar-locked` in both directions when crossing 1235px, and add a CSS rule to suppress `.lang-switcher-floating` opacity while locked.

### Changes
- `src/styles/global.css` — Added `html.sidebar-locked .lang-switcher-floating { opacity: 0; transition: none; }` after line 61. This forces the floating switcher to opacity 0 whenever `sidebar-locked` is active, preventing the flash regardless of what the media query or base CSS would otherwise set.
- `src/scripts/client.js` — Modified `handleMobileProfile()` shrinking branch (line 517-525): replaced `classList.remove('sidebar-locked')` with `classList.add('sidebar-locked')` + `setTimeout(300)` to remove it. This suppresses the floating switcher opacity during the sidebar slide-out animation.

### Build
- `npm run build` — Success, 515ms, 1 page built, no errors or warnings.

## Session 2: Fix mobile sidebar close — add slide-out transition

### Prompt
When clicking the X to close the mobile sidebar, it disappears instantly without animation. User wants the close transition to match the open slide-in animation.

### Plan
The CSS already defines `transition: transform 0.3s ease` on the mobile sidebar, and removing `.open` should trigger the reverse `translateX(-100%)` animation. The likely cause of the instant disappearance is a race condition: `closeSidebar()` is called from multiple event paths (overlay click, Escape, navigation, viewport resize), and if called twice in the same frame, the second call removes `.open` again before the browser paints the intermediate state, cancelling the transition.

Fix: (1) Add explicit `.sidebar:not(.open)` CSS rule to guarantee the close transition is always defined. (2) Add early return guard in `closeSidebar()` to prevent double-calls.

### Changes
- `src/styles/global.css` — Added `.sidebar:not(.open) { transform: translateX(-100%); transition: transform 0.3s ease; }` after `.sidebar.open` rule (line 1302). Ensures the close transition is explicitly defined regardless of other CSS rules.
- `src/scripts/client.js` — Added `if (!sidebar.classList.contains('open')) return` guard at the top of `closeSidebar()`. Prevents the race condition where multiple calls cancel the animation.

### Build
- `npm run build` — Success, 583ms, 1 page built, no errors or warnings.

## Session 3: Fix sidebar close — CSS cascade bug

### Prompt
Sidebar still disappears without animation when clicking X. Open animation works perfectly. Previous session's fix (`.sidebar:not(.open)` rule) didn't solve it.

### Root cause
The `.sidebar:not(.open)` rule added in session 2 has the **same specificity** (0,1,1) as `.sidebar.open`. Since it appears **later in source order**, it wins the cascade — even when `.open` IS present. This means `transform: translateX(-100%)` is always applied, and `.sidebar.open { transform: translateX(0) }` is dead code. The open "animation" that appeared to work was actually the `opacity: 1` from `.sidebar.open` (which has no competing rule). The transform never changed, so no transition ever fired in either direction.

### Changes
- `src/styles/global.css` — Removed the `.sidebar:not(.open)` rule entirely. The base mobile `.sidebar` rule already sets `transform: translateX(-100%); transition: transform 0.3s ease;`, and `.sidebar.open` (higher specificity) overrides the transform to `0`. Both open and close now trigger a style change → CSS transition fires.

### Build
- `npm run build` — Success, 519ms, 1 page built, no errors or warnings.

## Session 4: Fix sidebar close — opacity snapping to 0

### Prompt
Sidebar close animation still not working. Previous fix removed `.sidebar:not(.open)` rule but the sidebar still just disappears when clicking X.

### Root cause
The base desktop `.sidebar` rule (line 376) has `opacity: var(--sidebar-fade)`. On mobile, `--sidebar-fade = 0`. The mobile media query does NOT re-declare `opacity`, so the base `opacity: 0` persists. When `.open` is removed, opacity snaps from 1 to 0 instantly (not in the transition list), hiding the sidebar before the transform animation can be seen.

### Changes
- `src/styles/global.css` — Added `opacity: 1;` to the mobile `.sidebar` rule (line 1289). On mobile, the sidebar is a fixed off-screen drawer — visibility is controlled by `transform` only. The base `opacity: var(--sidebar-fade)` is irrelevant on mobile and was causing the close animation to be invisible.

### Build
- `npm run build` — Success, 509ms, 1 page built, no errors or warnings.
